About Project Page


GravedDamnit is a turn-based capture the flag strategy game, developed for the web gaming platform, that uses a wide variety of web services and technologies


It was our aim to create a strategy game, that will preserve the old feel and simplicity of traditional strategy games, but as well as give a interesting and enjoyable game to thos who are up for a challenge.

More about the game


GAMEPLAY: It is a 2-player strategy game(currently). Each player it's ranked on a certain level and it is assigned a specific MMR, according to the number of games lost or won. A bigger MMR is meant to get you in a match with a better and stronger player. Each player has a number of troops, with which he must fight and strategies to beat and steal the flag from the other player. The troops are of 4 types, each of them having different characteristics and purposes in the game. This gives the player a variety of strategies to choose from and enhances the gameplay diversity. Every troop has certain statistics, like atack damage or resistance which can be upgraded at according to the level of the player.

The troops are split into:

  1. Archers

  2. Tanks

  3. Runners

  4. Infantry

LOADOUTS: Loadouts are predefined sets, or sets defined by the player, of 6 troops(whichever type the player wants). Think of a deck of playing cards. This is a deck of troops. At the start of each game, each player could choose form an alreay existing loadout or make one of his own with the troops he has.

MAP: Being a strategy game, we considered the best option for a map would be a haxagonal one. It is essentially composed of the battlefield, which contains a wide variety of terrain from snow and woods to sand and lakes, and the two camps of player 1 and player 2. The map can as well be chosen just like the troops. We don't have a vast domain of choices, but we could see it in a further development

WINNING(when the game ends): The game ends when one the two players captures the others flag and manages to bring it to its own base, without loosing its own flag.

More about this

  • verb: the HTTP verb on which to match the route. Optional, the default is GET. Valid values are GET, POST, PUT, DELETE, etc. and the wildcard ANY which matches, obviously, any verb.

  • url: a string to match the route's URL. The URL has to be exactly equal to match. Optional, the default is an empty string. Must have a leading slash, as is the convention for HTTP URLs.

  • regex: a regex to match the route's URL. Optional, the default is an empty string. Take note that URLs have a leading slash.

  • target: the name of a file, relative to the site root, to be used be sent (or executed). This can be a directory, in which case /index is appended. Also, the extension can be missing, in which case .html, .htm, .pyml and .py are tried. Optional, the default is the URL (without a leading slash). This should not have a leading slash, since it is a relative path.

If both url and regex are missing (or default), the route matches any request (as long as the verb matches).

Note the following:

  • The space in Python strings did not terminate the expression, nor did spaces inside brackets.

  • The space delimiter after the Python expression in the first two paragraphs is kept.

  • The @ delimiter after the expression in the last two paragraphs is removed. This was used instead of a space because we do not want any spaces between the expression result and punctuation marks.

  • The @expression operator escapes characters relevant in HTML code so you're protected from injection. This is what you'll want to use most of the time.

  • In order to escape the whitespace between the format arguments, the entire expression was surrounded by brackets. This was not strictly necessary in this case, because the spaces in the "{}, {}" were escaped by " characters (' would work too), and the spaces in (city, country) were escaped by ( and ).

Database Side


Management: For the database management we used Oracle. It is a database consisting of 12 tables. Each one of them has characteristic constraints, triggers and indexes to handle large chunks of data. For example, the table "Player" was tested with over 10,000 rows. We have tested SQLInjection and ways to combat it and implemented CRUD (Create/Read/Update/Delete). Connection with the server was extablished using Python, more specifically, the cx_Oracle extension module. The scripts that access and manipulalte the database are called by Python controllers, that interact with the HTML and provide different functionalities.
